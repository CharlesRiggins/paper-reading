# 3. Experiments

## 3.1 Datasets

### Training Data
- **Part 1**: From Tulu-3 SFT data, partitioned by rules (multi-turn by turn, system messages separated from user messages, special delimiters). **23%** of samples were usable for block fine-tuning.
- **Part 2**: RAG samples — 20,000 each randomly sampled from `TriviaQA` and `2WikiMultiHopQA`. Each sample includes a question, 10 documents retrieved by Contriever, and an answer generated by `Llama3.3-70B-Instruct` based on those documents.

Training uses both full attention and block attention to preserve the model's full-attention capability.

### Evaluation Benchmarks

| Category | Benchmarks | Metric |
|----------|-----------|--------|
| **RAG** | Natural Questions, TriviaQA, HotpotQA, 2WikiMultiHopQA, NarrativeQA | Accuracy |
| **General** | MMLU, BBH, DROP, MATH, GSM8K, HumanEval, IFEval | Standard metrics |

Some general benchmarks are ICL tasks (exemplars become blocks); zero-shot tasks use full-attention mode directly.

## 3.2 Input Format

Documents are ordered by **ascending relevance score** (following Liu et al. [26]), with the format illustrated in Figure 3.

## 3.3 Base Model and Baselines

**Base model:** `Llama-3.1-Tulu-3-8B-SFT` (Tulu3-SFT)

| Model | Description |
|-------|------------|
| `Tulu3-RAG` | Full attention trained on RAG data (upper bound for RAG) |
| `Tulu3-RAG-promptCache` | PromptCache's training-free KV reuse [11] |
| `Tulu3-RAG-Superposition` | Superposition prompting parallel path [30] |
| **`Tulu3-block-ft`** | Block-attention model after block fine-tuning |
| `Tulu3-block-ft-full` | `Tulu3-block-ft` switched to full attention (verifying dual-mode) |
| `Tulu3-block-w/o-ft` | Direct Block-attention without fine-tuning (lower bound) |
| `Tulu3-block-w/o-pos` | Block fine-tuning without position re-encoding (ablation) |

## 3.4 Experimental Settings

- **Hardware**: 8 × NVIDIA H20 GPUs
- **Training**: learning rate $2 \times 10^{-5}$, batch size 64, 1 epoch, 20 warmup steps
- **Infrastructure**: DeepSpeed, Flash-Attention, bfloat16
- **Evaluation**: OpenCompass toolkit

## 3.5 Main Results

**Finding 1:** Direct switching to parallel prompt encoding methods (including Block-attention without fine-tuning) causes drastic accuracy drops. `Tulu3-block-w/o-ft` drops **16.2%** on average; PromptCache and Superposition degrade even more severely.

**Finding 2:** After block fine-tuning, Block-attention models perform comparably to full-attention models:
- RAG benchmarks: differences within **≤1%**
- ICL tasks: comparable or slightly higher

**Finding 3:** Block-attention and full-attention can seamlessly switch. `Tulu3-block-ft-full` is not inferior to the full-attention baseline on RAG and zero-shot tasks.

**Finding 4:** Position re-encoding is critical. Removing it causes an average accuracy drop of approximately **2%**, with occasional degradation.

Figure 4 shows that after approximately **800 training steps**, the model can switch between the two attention modes without loss.

## 3.6 Inference Efficiency

Assuming all retrieved document KV states are precomputed and cached, the improvements are dramatic:

| Total Length | TTFT Reduction | FLOPs-TFT Reduction |
|-------------|----------------|---------------------|
| 512 tokens | **48%** | **90.1%** |
| 32K tokens | **98.7%** | **99.8%** |

The first token outputs in only **45 ms** for 32K total input length. The longer the text, the greater the necessity of Block-attention.

## 3.7 Discussion

Block-attention enables developers to freely choose the optimal number of retrieval documents without worrying about inference latency. Its transformative potential extends far beyond RAG and ICL — particularly in game AI (Appendix A), where it can break through real-time performance bottlenecks.
